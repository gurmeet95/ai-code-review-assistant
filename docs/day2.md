## What we built: `CodeReview.java`

```java
package com.example.ai_code_review_assistant.entity;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Data;
import lombok.NoArgsConstructor;
import java.time.LocalDateTime;

@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
@Entity
@Table(name = "code_reviews")
public class CodeReview {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String fileName;

    @Lob
    @Column(columnDefinition = "TEXT")
    private String sourceCode;

    @Lob
    @Column(columnDefinition = "TEXT")
    private String resultJson;

    private Integer overallScore;

    private LocalDateTime createdAt;

    @PrePersist
    public void setCreatedAtOnPersist() {
        createdAt = LocalDateTime.now();
    }
}
```

## Annotations explained — what, how, why

### `@Data` (Lombok)
**What:** Generates, at compile time, getters for every field, setters for
every non-final field, `toString()`, `equals()`, and `hashCode()`.
**How:** Lombok is an annotation processor — it runs during compilation and
literally injects bytecode for these methods into the `.class` file. You never
see the generated code in your `.java` source, but it's really there (you can
confirm via IntelliJ's "Show Bytecode" or by decompiling the `.class` file).
**Why:** Without it, this class would need ~40+ lines of boilerplate
getters/setters we'd otherwise hand-write and maintain for zero real value.

### `@NoArgsConstructor` / `@AllArgsConstructor` (Lombok)
**What:** Generates a zero-argument constructor and a constructor taking every
field, respectively.
**How:** Same compile-time code generation as `@Data`.
**Why:** JPA/Hibernate **requires** a no-arg constructor on every `@Entity` —
it creates instances via reflection when loading rows from the database, and
reflection needs a no-arg constructor to call. `@AllArgsConstructor` is mainly
for convenience when constructing objects manually in code.

### `@Builder` (Lombok)
**What:** Generates a fluent builder: `CodeReview.builder().fileName("X").build()`.
**How:** Compile-time generation of an inner static `Builder` class.
**Why:** Cleaner than a long constructor call, especially as field count grows —
you can see which value maps to which field by name, and skip fields you don't
need to set explicitly (they default to `null`/`0`).

### `@Entity` (Jakarta Persistence / JPA)
**What:** Marks this class as one that Hibernate should map to a database table.
**How:** At startup, Spring Boot scans for `@Entity`-annotated classes and
registers them with Hibernate's metadata model. Combined with
`ddl-auto: update` in `application.yml`, Hibernate then auto-generates the
`CREATE TABLE` SQL for each one.
**Why:** This is the fundamental link between "a Java object" and "a database
row" — without it, JPA has no idea this class should be persisted at all.

### `@Table(name = "code_reviews")`
**What:** Explicitly sets the database table name.
**How:** Overrides Hibernate's default (which would just use the class name,
`CodeReview`, possibly with different casing depending on the DB dialect).
**Why:** Explicit is better than implicit — we don't want table names silently
depending on Hibernate's default-naming behavior; being explicit means the
name won't change if we ever swap databases or Hibernate versions change
defaults.

### `@Id`
**What:** Marks `id` as the primary key column.
**How:** Every `@Entity` must have exactly one field annotated `@Id` — this is
how Hibernate knows which column uniquely identifies a row (needed for
`UPDATE`/`DELETE`/relationship lookups).
**Why:** Non-negotiable requirement of JPA — an entity without a declared ID
won't even pass validation at startup.

### `@GeneratedValue(strategy = GenerationType.IDENTITY)`
**What:** Tells Hibernate the database itself auto-increments this column
(1, 2, 3…) — we never set `id` manually.
**How:** Maps to H2's native auto-increment column feature (seen in the
generated SQL: `id bigint generated by default as identity`).
**Why:** We want the database to guarantee uniqueness and ordering for us,
rather than generating IDs in application code (which risks collisions).

### `@Lob` + `@Column(columnDefinition = "TEXT")`
**What:** Marks `sourceCode` and `resultJson` as "Large OBject" fields, mapped
to a `TEXT` column instead of the default `VARCHAR(255)`.
**How:** Without `@Lob`/the explicit column definition, Hibernate would cap
these fields at 255 characters by default — far too small for actual Java
source code or a JSON review result.
**Why:** Uploaded code and Gemini's structured response can both be several KB
of text — `TEXT` has no practical size limit, `VARCHAR(255)` would silently
truncate or throw an error.

### `@PrePersist`
**What:** A JPA lifecycle callback — a method that Hibernate calls
automatically at a specific point in an entity's life. `@PrePersist`
specifically fires **once, right before the first `INSERT`** for a new entity.
**How:** Hibernate scans entity classes for lifecycle-annotated methods
(`@PrePersist`, `@PreUpdate`, `@PostLoad`, etc.) and invokes them at the
matching point automatically — no manual call needed anywhere in our code.
**Why:** We want `createdAt` set reliably, every time, without depending on
whoever constructs a `CodeReview` object (our future `ReviewService`) to
remember to set it manually. Moving that responsibility into the entity
itself means it's *impossible* to forget.

## Issues hit today

**1. `Incompatible types. Found: LocalDateTime, required: LocalDate`**
- Cause: field/import mismatch — `LocalDate` (date only) vs `LocalDateTime`
  (date + time) got mixed up while typing.
- Fix: confirmed both the field declaration and the import said
  `LocalDateTime` consistently.
- Lesson: `LocalDate` and `LocalDateTime` are genuinely different, unrelated
  types in `java.time` — always double check which one an IDE auto-import
  picked, since it's easy to accept the wrong suggestion.

**2. Naming convention slips:** `overAllScore` → should be `overallScore`,
`createdAT` → should be `createdAt`, `filename`/`sourcecode` → should be
`fileName`/`sourceCode`.
- Lesson: Java camelCase capitalizes only the first letter of each new word —
  acronym-like endings don't get fully capitalized either. This matters
  beyond style because Lombok generates getter/setter names directly from
  field names — inconsistent casing here propagates into inconsistent method
  names everywhere the class is used.

**3. Missing `private` modifier on one field (`resultJson`).**
- Lesson: package-private fields can be accessed directly by any class in the
  same package, bypassing `@Data`'s generated getters/setters. `private`
  should be the default for every entity field, no exceptions.

## Verified

Ran the app and confirmed in the console log:
```
Hibernate: create table code_reviews (id bigint generated by default as identity,
created_at timestamp(6), file_name varchar(255), overall_score integer,
result_json TEXT, source_code TEXT, primary key (id))
```
Also confirmed via H2 console: `SHOW TABLES;` lists `CODE_REVIEWS`.

This confirms the full chain works: Java `@Entity` annotations →
Hibernate DDL generation → actual table in H2, with correct column types
(`TEXT` for the `@Lob` fields, `timestamp(6)` for `LocalDateTime`, camelCase
Java fields auto-converted to snake_case SQL columns).
